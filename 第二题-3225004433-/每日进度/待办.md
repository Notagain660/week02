成功时返回了 User 对象，但前端可能更想要新报修单的 orderId（这样可以在页面显示“报修单号xxx已创建”）。你可以让 Service 返回 orderId，然后 return MapperResult.success(orderId)。

返回类型写了 MapperResult（原始类型），建议写成 MapperResult<Object> 或 MapperResult<Long>，这样更明确。

4. 写 `UserMapper.xml`（里面写具体的SQL语句）

## 🧩 用C语言的思路理解Java的分层

假设你要用C语言写一个宿舍报修系统，你会怎么做？

1. **定义数据结构**：写一个 `struct user`（对应 Java 的 `User` 类）。
    
2. **写操作函数**：比如 `void save_user(struct user u)`，里面用文件操作把数据存到 `user.txt`（对应 Java 里 `insertUser` 方法，但存到数据库）。
    
3. **写业务逻辑函数**：比如 `void register()`，里面调用 `save_user`（对应 Java 的 Service 层）。
    
4. **写主函数**：循环打印菜单，调用 `register`（对应 Java 的 UI 层）。
    

那么问题来了：

- 如果以后想换存储方式（比如从文件换成数据库），你得重写 `save_user` 函数。
    
- 如果以后想加一个功能（比如查询用户），你得修改很多地方。
    

为了解决这些问题，Java 项目里把**函数按职责分到不同的文件/包**：

- **entity**：放 `struct` 定义。
    
- **mapper**：放所有直接操作数据库的函数（类似C里专门操作文件的函数集合）。
    
- **service**：放业务逻辑（比如注册时检查用户名是否重复、密码加密），它会调用 mapper 里的函数。
    
- **ui**：放主函数和菜单交互。
    

这样，哪天你想从文件换到数据库，只需要改 mapper 里的实现（或者从手写文件改为用MyBatis），其他层完全不用动。

## 🛠️ MyBatis 在 C语言里相当于什么？

MyBatis 其实就是帮你**省去了手动写数据库连接、执行SQL、处理结果集的重复代码**。在C语言里，如果你要操作数据库，你可能需要写很多类似的代码（连接、发送SQL、解析返回）。MyBatis 把这些都封装好了。

类比：

- 在C里你直接写 `fopen`、`fprintf` 等，这是“手写文件操作”。
    
- 用 MyBatis 就像你用一个现成的“文件操作库”，你只需要告诉它“把这条数据存到表里”，它自动帮你完成打开文件、写入、关闭等操作。

`n -> repairorderservice.loginRepairorder(n)` 等价于方法引用 `repairorderservice::loginRepairorder`。

- 当 ==lambda== 表达式的参数直接传递给一个已有方法时，就可以使用方法引用。
    
- 前提是 `loginRepairorder` 方法的签名是 `boolean loginRepairorder(Long orderId)`，这与 `LongPredicate` 的 `boolean test(long value)` 兼容（Java 会自动将 `long` 装箱为 `Long`）。

        repairorderservice::loginRepairorder,   // 方法引用
        "报修单号 %d 不存在，请重新输入：");

如果你需要取反（校验不存在），则无法直接使用方法引用，因为 `!` 不是方法的一部分。你可以继续保持 lambda 或创建一个专门的方法，例如 `notExists`。

1. **`@Service` 放错了位置**
    
    - `@Service` 必须放在 **类声明** 的上方，而不是方法上。
        
    - 正确：`@Service public class UserService { ... }`
        
2. **`@Autowired` 不能用在方法内部的局部变量**
    
    - `@Autowired` 只能用在类的成员变量、构造器或 setter 方法上。
        
    - 你需要将 `private UserMapper userMapper;` 声明为 **类的成员变量**，并在其上方加 `@Autowired`。
- 成员变量就是定义在类里面、方法外面的变量。它属于对象（实例变量）或类（静态变量）。mapper:注入为成员变量


 Spring Boot 中，我们不再手动创建 `SqlSessionFactory`，而是让 Spring 帮我们把 `UserMapper` 注入到 `UserService` 里。注入的方式就是：

1. 在 `UserService` 类里**声明一个成员变量**，类型是 `UserMapper`。
    
2. 在这个成员变量上加上 `@Autowired` 注解。
    
3. 在类上加上 `@Service` 注解，告诉 Spring 这个类需要被管理。


`@RequestMapping` 和 `@GetMapping` 都是 Spring MVC 中用于将 HTTP 请求映射到处理器方法的注解，它们的主要区别在于：

- **`@RequestMapping`** 是一个通用的映射注解，可以处理任意 HTTP 方法（GET、POST、PUT、DELETE 等）。使用时需要显式指定 `method` 属性，例如 `@RequestMapping(value = "/user", method = RequestMethod.GET)`。
    
- **`@GetMapping`** 是 `@RequestMapping(method = RequestMethod.GET)` 的快捷方式，专门用于处理 GET 请求，代码更简洁。
