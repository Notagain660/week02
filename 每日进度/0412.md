##### 发帖功能：一个 HTTP 请求只能有一种 `Content-Type`：
- 如果用 `multipart/form-data`（支持文件上传），整个请求体都是 multipart 格式，无法同时用 `@RequestBody` 解析整个JSON请求体。
- 如果用 `application/json`，可以方便地用 `@RequestBody` 接收对象，但无法直接上传文件。
- 先上传图片，获取 URL，再发帖时只传 URL（更常见），即前端完全可以在一个表单里让用户同时填写文本和选择图片，但在发送请求时**分两步**：
    1. **先调图片上传接口**：把用户选的图片文件上传到服务器，服务器返回一个图片 URL。（跟头像修改的getavatar套路一致）
    2. **再调发帖接口**：把用户填的文本信息（如标题、内容、地点等）以及上一步拿到的图片 URL，一起用 `@RequestBody` 以 JSON 格式发送到后端。
- ai辅助描述（先写成一定执行的）：aiservice，提示词+chatmodel得到回应+结果、输出、转文本
- 用户不能设置发送者id（后端从当前用户这个线程拿）、状态、是否置顶（默认不置顶）、发布时间
- itemtime（捡到/丢失物品的时间）从localdate -> localdatetime（不可能让用户输入精确到秒甚至毫秒的时间）

#### 接口在事务内完成：
**事务**：一组数据库操作（比如插入帖子、更新用户发帖数）要么全部成功，要么全部失败，不会出现部分成功部分失败的情况。
**为什么接口要在事务内完成？**  
一个业务接口往往涉及多张表的多次写操作。例如“发帖”：
- 插入 `post` 表
- 更新 `user` 表中的 `userpostnum` +1  
    - 如果不使用事务，可能插入帖子成功但更新用户数量失败，导致数据不一致。
**Spring Boot 中的实现**：
- 在 Service 层方法上加上 `@Transactional` 注解（来自 `org.springframework.transaction.annotation.Transactional`）。
- 方法内所有数据库操作会共用一个数据库连接，一旦抛出未捕获的异常（默认 `RuntimeException`），自动回滚。
- 示例：
````{
@Transactional
    public void createPost(PostDTO dto) {
        postMapper.insert(post);
        userMapper.incrementPostCount(dto.getUserId());
    }
}
````
- 接口（Controller）通常不直接加事务，而是调用 Service 层，事务边界在 Service 层方法上。
> ⚠️ 注意：事务对查询操作（SELECT）不是必须的，但若查询后需要加锁（如防止超卖），可设置 `@Transactional(isolation = ...)`。
